# SDD 개발

## 개요

`sdd/`를 공식 산출물 기록 시스템으로 사용하는 저장소의 구현 작업에 이 스킬을 사용합니다.

이 스킬은 다음 하나의 워크플로우를 강제합니다:
1. 관련 `sdd/01_planning` 산출물을 먼저 검토하고 수정합니다.
2. `sdd/02_plan/<섹션>/` 아래에 태스크 계획을 생성하거나 갱신합니다.
3. 코드 작업을 수행합니다.
4. `sdd/03_build`에 현재 구현 요약을 기록합니다.
5. `sdd/04_verify`에 현재 유지 검증 요약을 기록합니다.
6. 롤아웃 시 `sdd/05_operate`에 배포 및 모니터링 결과를 기록합니다.

롤아웃이 명시적으로 범위에 포함되고 저장소에 DEV와 PROD 환경이 분리되어 있는 경우, 이 스킬은 단계적 릴리스 규칙을 강제합니다: DEV에 먼저 배포하고, 전체 레이어 검증을 완료한 후, DEV 통과 시에만 PROD로 승격하고, PROD에서 동일한 검증을 재실행합니다. PROD 검증이 실패하면, 사용자가 명시적으로 방향을 변경하고 해당 위험이 기록되지 않는 한 롤백이 필요합니다.

영속성에 영향을 미치는 작업의 경우, 이 스킬은 스키마 동등성 검증도 강제합니다. 배포된 실제 상태가 코드와 일치한다고 가정하지 말고, 영향받는 데이터베이스 객체의 실제 DEV 및 PROD 스키마 상태와 마이그레이션 또는 모델 의도를 항상 비교합니다.

사용자가 명시적으로 금지하지 않는 한, 코드 편집 상태로는 작업이 완료되지 않습니다. 일관된 git 커밋으로 마무리하고, 유지된 결과를 설정된 원격 저장소에 푸시하며, 저장소가 설치 가능한 패키지나 CLI를 제공하는 경우 새 버전을 배포하고 레지스트리에 반영되었는지 확인합니다.

`sdd/` 내부의 정확한 경로가 필요할 때는 [references/section-map.md](references/section-map.md)를 읽습니다.

화면 스펙 기반 UI 작업의 경우, 스펙의 재사용 가능한 정적 에셋은 코드에 사용하기 전에 저장소의 공식 에셋 빌더를 통해 추출해야 합니다.

화면 스펙 기반 레이아웃 작업의 경우, 공식 디자인 가이드 빌더가 있다면 먼저 검토합니다.

로컬 화면 정확도를 위해, 저장소의 Playwright 정확도 러너와 스위트 레지스트리가 있을 경우 이를 공식 자동화 게이트로 취급합니다.

검증 작업의 경우, 회귀 범위 선택을 필수 유지 산출물로 취급하고 `sdd/02_plan`, `sdd/03_build`, `sdd/04_verify`를 통해 이를 이어갑니다.

`sdd/05_operate`의 존재만으로 롤아웃 범위를 추론하지 않습니다; 사용자가 배포를 요청하거나 저장소의 현재 정책/계획이 완료의 일부로 롤아웃을 포함할 때만 롤아웃을 요구합니다.

저장소나 팀이 `main`에서의 DEV 배포를 완료 기준으로 삼는 경우, 임시 브랜치나 워크트리는 작업 공간일 뿐입니다. 태스크가 배포되었다고 선언하기 전에, 최종 유지 변경 사항을 `main`에 반영하고 `origin/main`을 푸시합니다.

## 사용 시기

다음 경우에 이 스킬을 사용합니다:
- 저장소에 `sdd/01_planning`, `02_plan`, `03_build`, `04_verify`, `05_operate`가 있을 때,
- 사용자가 `개발해`, `작업해`, `구현해`, `수정해`, `고쳐`, `리팩토링해`, `테스트해`, `배포해` 같은 개발 지시를 내릴 때,
- 사용자가 `화면명세서`, `화면설계서`, `화면 설계`, `화면`, `화면 스펙`, `UI`, `디자인`, `디자인 가이드`, `screen spec`, `screen design`, `design guide` 같은 화면/UI 작업을 요청할 때,
- 사용자가 해당 폴더를 통해 작업을 추적 가능하게 하길 원할 때,
- 태스크에 구현과 문서화/검증 업데이트가 모두 포함될 때,
- 태스크가 배포 또는 운영 후속 작업으로 끝날 수 있을 때.

다음 경우에는 이 스킬을 사용하지 않습니다:
- 저장소 변경이 없는 일상적인 질문,
- 영속적인 SDD 기록이 필요 없는 일회성 로컬 디버깅,
- `sdd/`를 주요 문서 시스템으로 사용하지 않는 저장소.

## 워크플로우

### 1. 먼저 Planning 검토

- 코드를 편집하기 전에 영향받는 계획 영역을 파악합니다.
- `sdd/01_planning`에서 관련 산출물만 엽니다:
  - feature (기능)
  - screen (화면)
  - architecture (아키텍처)
  - data (데이터)
  - api
  - iac
  - integration (통합)
  - nonfunctional (비기능)
  - security (보안)
  - test (테스트)
- 구현이 이미 계획에서 벗어났다면, 코딩 전에 계획 산출물을 먼저 수정하거나 최소한 이탈 사항을 기록합니다.

### 2. 계획 생성 또는 갱신

- `sdd/02_plan/<섹션>/` 아래에 영속적인 계획 파일을 생성하거나 재사용합니다.
- 저장소의 계획 스캐폴드가 있으면 우선 사용합니다.
- 계획에는 다음이 포함되어야 합니다:
  - 범위 (scope)
  - 가정 (assumptions)
  - 수락 기준 (acceptance criteria)
  - 실행 체크리스트 (execution checklist)
  - 현재 노트 (current notes)
  - 검증 (validation)
- 배포로 끝날 수 있는 태스크의 경우, 수락 기준에 DEV 게이트, 대응하는 PROD 게이트, 유지되는 전체 레이어 테스트 표면, 롤백 트리거/경로가 명시적으로 포함되어야 합니다.
- 영속성, 모델, 리포지토리, 마이그레이션, SQL, ORM 매핑, 또는 스키마 드리프트와 관련될 수 있는 런타임 실패에 영향을 미치는 태스크의 경우, 수락 기준에 DEV/PROD 스키마 검증이 명시적으로 포함되어야 합니다.
- 체크리스트에서 정확히 하나의 항목만 진행 중 상태로 유지합니다.

### 3. 계획에 따라 구현

- 영향받는 계획 산출물과 계획이 충분히 정렬된 후에만 코드를 변경합니다.
- 의미 있는 편집이나 결정 후에는 계획 현재 노트를 갱신합니다.
- 문서 생성기나 캡처 파이프라인이 관련된 경우, 해당 도구를 `sdd/99_toolchain` 아래에 유지합니다.
- 로컬 화면 정확도를 위해, 저장소가 해당 표면에 대한 더 강력한 정확도 게이트를 문서화하지 않는 한 Playwright를 공식 자동화 게이트로 취급합니다.
  - 이 템플릿으로 만들어진 저장소에서는 임시 `npx playwright test ...` 대신 `python3 sdd/99_toolchain/01_automation/run_playwright_exactness.py --suite <스위트-id>`를 우선 사용합니다.
  - 스위트 레지스트리를 `sdd/99_toolchain/01_automation/playwright_exactness_manifest.py`에 유지합니다.
  - Browser Use, 수동 스크린샷, 시맨틱 추출은 진단을 보완할 수 있지만, 스위트가 존재하는 경우 유지되는 Playwright 정확도 게이트를 대체하지 않습니다.
  - 필요한 스위트가 아직 없다면, 동일 태스크 내에서 추가하거나 확장하고, 작업 완료 전에 등록합니다.
- 태스크가 화면 스펙에 보이는 아이콘, 로고, 일러스트레이션 또는 기타 정적 에셋에 의존하는 경우, 저장소의 공식 Asset Spec Builder를 먼저 사용합니다.
  - 이 템플릿으로 만들어진 저장소에서는 보통 `sdd/99_toolchain/01_automation/spec_asset_builder.py` 또는 그 래퍼/매니페스트입니다.
  - 재사용 가능한 에셋 계획 기록은 `sdd/01_planning/02_screen/assets/` 아래에 둡니다.
  - 수작업 추적이나 스크린샷 크롭 대신, 승인된 PDF/이미지 소스에서 런타임 에셋을 빌드합니다.
  - 에셋이 소스 크롭과 정확히 일치해야 하는 경우 `--verify-exact` 같은 정확도 검증을 사용합니다.
  - 소스, 매니페스트, 생성된 에셋 경로, 예외 사항을 `sdd/03_build`와 `sdd/04_verify`에 기록합니다.
  - 빌더가 에셋을 표현할 수 없는 경우에만 수동 재작성을 대안으로 사용하고, 계획/빌드/검증 기록에 해당 예외를 명시적으로 문서화합니다.
- 태스크가 화면 스펙에서 파생된 간격, 레이아웃 밀도, 타이포그래피, 색상 리듬, 또는 컴포넌트 계층에 의존하는 경우, 저장소의 공식 디자인 가이드 빌더를 먼저 검토합니다.
  - 이 템플릿으로 만들어진 저장소에서는 `sdd/99_toolchain/01_automation/README.md`를 검토하고 저장소에 실제로 존재하는 빌더, 래퍼, 또는 매니페스트를 사용합니다.
  - 수동 간격이나 팔레트 조정 전에 생성된 가이드를 작업 기준으로 사용합니다.
  - 저장소에 아직 디자인 가이드 빌더가 없다면, 수동 해석 소스를 계획/빌드/검증에 문서화합니다.
- 구현 완료를 선언하기 전에 회귀 표면을 정의합니다.
  - `sdd/02_plan/10_test/regression_verification.md`에서 시작합니다.
  - 직접 대상과 변경 사항의 영향을 받는 상위, 하위, 공유 표면을 파악합니다.
  - 변경 사항이 공유 라우팅, 셸/인증, 공유 상태, 공통 컴포넌트, 계약, 생성된 에셋, 또는 빌더 출력에 영향을 미치는 경우, 편집된 모듈만 검증하는 대신 회귀 범위를 넓힙니다.
  - 선택된 회귀 범위와 정당화된 제외 사항을 계획/빌드/검증에 기록합니다.
- 롤아웃이 범위에 포함된 경우, 하나의 유지되는 전체 레이어 검증 표면을 정의하고 DEV와 PROD 모두에서 재사용합니다.
  - 변경 사항의 영향을 받는 관련 앱/런타임 진입점, API/계약, 영속성/스키마, 작업 또는 워크플로우 부작용, 공유 통합, 상태/모니터링 검사를 포함합니다.
  - 사용자가 명시적으로 그 예외를 승인하고 기록되지 않는 한, DEV 게이트보다 좁은 검증 표면으로 PROD에 승격하지 않습니다.

### 4. 빌드 요약 기록

- `sdd/03_build`에 구현한 내용을 기록합니다.
- 다음을 사용합니다:
  - `03_build/01_feature`: 기능 구현 요약
  - `03_build/02_screen`: 화면 구현 요약
  - `03_build/03_architecture`, `06_iac`, `10_test`: 현재 상태 횡단 요약
- 항목은 사실 기반으로, 현재 상태만 기록합니다: 구현 범위, 모듈, 에셋, 계약, 현재 사용자 가시 동작.

### 5. 검증 기록

- `sdd/04_verify`에 유지되는 검증 상태를 기록합니다.
- 다음을 사용합니다:
  - `04_verify/01_feature`: 기능 검증 요약
  - `04_verify/02_screen`: 화면 검증 요약
  - `04_verify/03_architecture`, `06_iac`, `10_test`: 현재 유지 검사 및 잔존 위험
- 커맨드 수준의 검증 증거 없이 완료를 선언하지 않습니다.
- 단계적 DEV → PROD 롤아웃의 경우, 검증은 두 환경 모두에서 동일한 유지되는 전체 레이어 검증 표면을 사용해야 합니다.
  - 배포 후 DEV에서 전체 레이어 검증을 실행하고, PROD 승격 전 하드 게이트로 취급합니다.
  - PROD 배포 후, PROD에서 동일한 유지 검증 표면을 재실행합니다.
  - PROD 검증이 실패하면, 즉시 롤백 또는 승인된 복구 절차를 실행하고 실패 및 복구 결과를 기록합니다.
- 영속성에 영향을 미치는 작업의 경우, 검증에는 해당 환경이 존재할 때 DEV와 PROD 모두의 실제 스키마 증거가 포함되어야 합니다.
  - 마이그레이션 상태와 실제 런타임 스키마를 별도로 확인합니다.
  - 변경 사항이 영향을 미치는 테이블, 컬럼, 인덱스, 제약 조건, 트리거, 기본값, 레거시 호환성 객체를 검증합니다.
  - 사용된 커맨드나 쿼리, 확인된 환경, 드리프트 또는 동등성 결과를 기록합니다.
- 회귀 검증은 필수입니다.
  - 검증은 직접 표면과 회귀 기준에서 선택된 유지되는 상위/하위/공유 표면을 포함해야 합니다.
  - 필요한 회귀 슬라이스에 대한 자동화가 아직 없다면, 가능한 최선의 수동 또는 커맨드 검사를 실행하고 해당 격차를 현재 잔존 위험으로 기록합니다.
  - 해당 표면에 Playwright 스위트가 존재하는 경우, 공식 러너 커맨드, 스위트 id, 스크린샷/json 산출물 경로, 라이브 대 로컬 분리를 명시적으로 기록합니다.

### 6. 운영 결과 기록

- 배포 또는 런타임 후속 작업이 발생한 경우, `sdd/05_operate`를 갱신합니다.
- 다음을 사용합니다:
  - `05_operate/01_runbooks`: 영속적인 운영 절차 변경 사항
  - `05_operate/02_delivery_status`: 현재 라이브 상태 및 모니터링 기준
- 다음을 기록합니다:
  - 무엇이 배포되었는지
  - 현재 어떤 라이브 기준이 적용되는지
  - 어떻게 모니터링되는지
  - 현재 잔존 위험
- 단계적 DEV → PROD 롤아웃의 경우, DEV 게이트, PROD 게이트, 롤백 결과를 명시적으로 기록합니다.
- DEV 배포 기준이 `main`에 연결되어 있는 경우, 사이드 브랜치 푸시를 충분한 것으로 취급하지 않습니다. 최종 변경 사항을 `main`에 머지, 체리픽, 또는 재적용하고, `origin/main`을 푸시한 후, 해당 `main` 기준에 대한 배포 증거를 기록합니다.

## 가드레일

- `sdd/`가 있을 때 병렬 `docs/` 트리를 생성하거나 재구성하지 않습니다.
- 코드 변경이 작아 보인다는 이유로 계획 검토를 건너뛰지 않습니다.
- 빌드, 검증, 운영 증거를 `sdd/`에 남겨야 할 때 채팅 텍스트에만 남기지 않습니다.
- 영속적인 변경이 남아있을 때 로컬 편집 후 멈추지 않습니다; 사용자가 명시적으로 금지하지 않는 한 커밋과 푸시가 기본 완료 기준입니다.
- 배포된 동작이나 지침을 변경한 후, 사용자가 명시적으로 배포를 금지하지 않는 한 배포 가능한 패키지나 CLI를 미배포 상태로 두지 않습니다.
- PROD 롤아웃이 범위에 포함된 경우, 유지되는 전체 레이어 DEV 검증 표면이 통과하기 전에 PROD로 승격하지 않습니다.
- PROD 롤아웃이 범위에 포함된 경우, 사용자가 명시적으로 승인하고 위험이 기록되지 않는 한 DEV를 게이팅한 것보다 약한 PROD 검증 표면을 사용하지 않습니다.
- PROD 롤아웃이 범위에 포함된 경우, "PROD 배포 성공"에서 멈추지 않습니다; 배포 후 PROD 검증은 필수입니다.
- PROD 롤아웃이 범위에 포함된 경우, 실패한 PROD 배포를 미해결 상태로 두지 않습니다; 롤백 또는 승인된 복구 절차를 실행하고 기록해야 합니다.
- 로컬 테스트, 마이그레이션 헤드, 또는 현재 모델 코드가 배포된 스키마 동등성을 증명한다고 가정하지 않습니다.
- 스키마 드리프트가 동작에 영향을 미칠 수 있는 영속성 영향 작업에 대해 DEV/PROD 스키마 검사를 건너뛰지 않습니다.
- 저장소에 공식 Asset Spec Builder가 있을 때 화면 스펙 정적 에셋을 수동으로 다시 그리지 않습니다; 먼저 추출하고 생성된 에셋을 사용합니다.
- 요청된 UI 변경이 작은 수동 조정처럼 보인다는 이유로 관련 화면 자동화 빌더를 건너뛰지 않습니다.
- 변경 사항이 공유 또는 인접 동작에 영향을 미칠 수 있을 때 편집된 파일, 라우트, 또는 화면에서 검증을 멈추지 않습니다.
- 유지되는 SDD 기록에서 회귀 범위 선택을 생략하지 않습니다.
- 롤아웃에 도달하지 못한 태스크에 대해 `05_operate`를 갱신하지 않습니다; 대신 롤아웃이 발생하지 않았음을 명시적으로 기록합니다.
- 저장소나 팀 기준이 배포된 변경 사항이 `origin/main`에 있기를 기대할 때 임시 브랜치에서의 DEV 롤아웃을 완료된 것으로 선언하지 않습니다.

## 산출물 기준

구현 태스크가 끝날 때 기대되는 기록:
- 계획 산출물 검토 또는 수정 완료,
- 계획 파일 갱신 완료,
- 빌드 요약 작성 완료,
- 선택된 회귀 범위 및 잔존 위험을 포함한 검증 요약 작성 완료,
- 롤아웃이 발생한 경우 운영 상태 갱신 완료,
- 일관된 git 커밋 생성 및 푸시 완료,
- 배포 가능한 패키지/CLI인 경우, 사용자가 명시적으로 금지하지 않는 한 레지스트리 배포 및 최신 버전 확인 완료.

PROD 롤아웃이 범위에 포함된 경우, 유지되는 완료 상태에는 다음도 필요합니다:
- DEV 배포가 먼저 이루어지고 유지되는 전체 레이어 DEV 검증 표면 통과,
- PROD에서 동일한 유지되는 전체 레이어 검증 표면 재실행,
- 스키마가 동작에 영향을 미칠 수 있는 경우 DEV/PROD 스키마 상태 확인 및 기록,
- PROD 검증 실패 시 verify/operate에 롤백 또는 복구 증거 기록.
